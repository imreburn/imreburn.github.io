Review Instruction

  • The review should summarize
    • what the paper is about
    • whether the problem is important
    • list strengths and weaknesses of the approach they use to solve it
    • whether their experiments are a good demonstration of method success

From the Title

  • Original title: Edge-cloud orchestration of assertion-based monitors for robotic applications
    • edge-cloud : distributed computing architecture Edge Computing
    • orchestration : allocating monitors
  • Interpreting the title: Dynamic allocation of runtime monitors over distributed network of robotic systems
    • Allocation problem : Migrating a monitor to another edge (== layer, computing node) when necessary (ex. CPU shortage)
      • Goals : Maximize verification responsiveness, accuracy
      • Constraints : CPU resource

Importance

  • I think the problem is important because 1) normal operation of edge devices can be affected if runtime monitor takes too much computational resource, 2) CPU usage is fluctuating while executing mission, and 3) a lagging control loop of a cyber-physical system could result in delayed responses to physical environment.
  • Also additional handling would be needed when runtime monitor requires computational resources more than the entire network's resource.

Questions (before reading main sections)

  • Overhead : How much does balancing cost?
    • Computing resource
    • Communication (network)
    • Space
    • Any other trade-offs?
  • Performance of allocating monitors
  • Scalability in terms of # of assertions, nodes
  • How do authors handle heterogeneity?
    • Computing nodes can have different hardware and software

Strength

  • The paper covers realistic scenarios such as overloading all CPUs and proposes the best possible solution with minimal trade-off

Strength - Low communication overhead by choosing ROS

  • Each machine can easily subscribe a topic
  • Migrating a monitor does not require sending the entire monitor; old node just cancels the subscription of the topic and new node starts the subscription
  • Old node sends gap of data only after receiving the timestamp when new node starts the subscription

Strength - Evaluation

  • The paper evaluates two case studies. Both are well designed to show the performance of allocating monitors. Specifically, authors provide realistic scenarios where CPU usages of computing nodes are fluctuating, and sometimes are saturated.
  • The configurations and procedures are thoroughly explained.
  • Case study 2 involves real-world robotic systems; it is effective to show whether the concept is applicable to real-world platforms

Weakness

Weakness - Scalability

"the orchestrator spent on average 27ms (every second) to solve the MILP problem, proving the scalability of this approach"

  • Showing a runtime overhead for a single case is not enough to prove the scalability
  • Beyond the scalability of the allocation problem, it overall lacks in demonstrating how it is scalable depending on the number of assertions/nodes.

Weakness - Space overhead

"Each machine has a copy of all monitors and each monitor is containerized through Docker"

"We also observed a negligible and static increase of ≈ 40MB of system memory usage on all nodes of the cluster due to the allocation of the resources needed by the monitors"

  • I think space overhead is not discussed enough.
  • Space overhead caused by containerization or scalability of space overhead are not shown in the paper.
  • The fact that each node has to possess all monitors might be a disadvantage because a device with limited computing power might also have limited space.
  • I think growth of events buffer is not discussed enough.

Other Questions & Comments

Allocation problem (MILP), Eq (2)

  • MILP is generally known as NP-hard : is it scalable?
  • How frequent is MILP solved?

    "The orchestrator is set to recompute the optimal allocation of monitors every second."

    • Can we solve the allocation problem only if necessary (eg. CPU saturation), instead of every second?
  • I am confused with the term "availableCPU"

    From Eq (2) : " percentage of unused CPU plus the CPU consumption of the verification process on computing unit ;"

  • What is "CPU consumption of verification process"?
  • How is it different from ?
  • Why is "CPU consumption of verification process" not a variable?
  • Is this "plus" correct here?
  • I suggest explaining functions ( and others) first before showing Eq (2)

CPU Overhead

  • When comparing to "no handler" scenario, it looks each assertion takes CPU usage more than I guess (Fig. 7 and Fig. 9)
    • (Fig. 7) 50 monitors takes approximately 40% of CPU of edge node (or 18% of server node; it depends on configurations)
    • It means that each monitor takes ~0.8% CPU usage of edge node : Is it good or reasonable?
  • CPU usage for orchestration (MILP problem) is not shown in both case studies. Is it negligible?

Assuming that overhead of assertion is constant

"All topics are published on the edge machine, which represents the most common configuration in most practical use cases."

  • No complex assertion - where topics from multiple nodes are needed - is considered in the evaluation
  • In consequence, the paper treats all assertions are equivalent in terms of complexity; it assumes that each assertion requires the same (or almost same) amount of computing resource

The point of using Signal Temporal Logic (STL)?

The paper adopts STL for translating assertions. I understand it fits for CPS, but I am not sure why this is emphasized. I had following questions when reading intro but they are not answered.

  • Is this new work to use STL for assertions?
  • STL is especially good for this situation?

Time synchronization

"Every computing node in our architecture is synchronised with the precision time protocol (PTP) guaranteeing synchronisation errors of the order of nanoseconds"

  • Migrating runtime monitors require time synchronization of nodes in the network
  • Why do we need time synchronization?
    • Time synchronization is required if the timestamp is created when a node receives a subscribed event.
    • Time synchronization would not be needed if the timestamp is created when an event occurs and the message includes both the value and the timestamp

Appendix

Architecture

Assumption

  • A fully connected network with ROS-compliant nodes which are time synchronized

Architecture

  • Monitor
    • A monitor is synthesized from an assertion
      • 1 monitor per assertion
    • events buffer -> sorted buffer -> compressed boolean buffer
    • Each event is stored as (topic, value, timestamp)
    • Events buffer is sorted when the sorting threshold is met
    • Timestamp is removed in the compressed boolean buffer
Question

Is it okay to remove timestamps, only preserve the order? How can we handle STL intervals?

  • Monitor handler
    • 1 monitor handler per node
    • Stat handler : send statistics (ex. CPU usage) to the coordinator
    • Request handler : handle the request (ex. MIGRATE) from the coordinator
    • Scheduler : schedule executing monitors based on priority
  • Global coordinator
    • 1 coordinator for the network : placed at one of nodes with the highest available resource
    • Solves the allocation problem
